== 重点

. 复习一下大O,用来表示算法复杂度，算法复杂度(具体又分为时/空间复杂度，最差情况，最好情况等)与性能一般(指整体来看大部分场景，具体还是以真实测量为准)呈反比
.. 复杂度升序(越靠前性能越好): stem:[O(1) < O(log_2n) < O(sqrt(n)) < O(n) < O(nlog_2n) < O(n^2) < O(2^n) < O(n!)]
.. 前面也说了，具体要看实际场景，算法在不同的场景表现并不相同
.. 提到了动态类型的成本，单一操作很可能比长期大量操作花费比平均值更长的时间，因为存在需要动态分配内存的操作
.. 不同复杂度下，操作的元素个数越多，对耗时的影响越明显，这也说明了一个点，即要根据自己的场景，预估操作的元素的数量级，来选择对应类型和数据结构，不是说完全按算法复杂度来选，不能太死板，思路要开阔和灵活(比如综合考虑易用性，可读性，扩展之类)
